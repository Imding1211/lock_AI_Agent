# 系統維運與資料管理手冊

本手冊旨在指導維運人員如何管理系統持久化資料（SQLite 與 User Profiles），以及在生產環境中進行故障排查。

---

## 1. 資料庫管理 (SQLite)

系統使用 `chat_history.db` 存儲所有使用者的對話狀態與 Checkpoints。

### 1.1 資料庫路徑
預設位於專案根目錄下的 `chat_history.db`（可透過 `config.toml` 的 `[memory].path` 調整）。

### 1.2 資料備份與清理
*   **備份**：由於 SQLite 是單一檔案，直接複製該檔案即可完成備份。建議每日定時備份。
*   **清理**：
    *   **全量清理**：刪除該檔案後重啟服務，系統會自動重新建立。這會導致所有使用者失憶。
    *   **特定用戶清理**：若需重置特定使用者的記憶，需進入資料庫執行：
        ```sql
        DELETE FROM checkpoints WHERE thread_id = 'smart_lock_{user_id}';
        DELETE FROM checkpoint_writes WHERE thread_id = 'smart_lock_{user_id}';
        ```

### 1.3 注意事項
*   由於使用 `aiosqlite` 並開啟了 `check_same_thread=False`，在高併發下請觀察磁碟 I/O。
*   如果資料庫檔案異常增大（例如超過 500MB），請檢查 `max_messages_threshold` 是否設定過大，導致壓縮頻率不足。

## 2. 使用者輪廓管理 (User Profiles)

輪廓以 Markdown 格式儲存，方便人工閱讀與微調。

### 2.1 儲存路徑
位於 `user_profiles/` 資料夾下，檔名為 `{user_id}.md`。

### 2.2 手動修正
若 AI 萃取了錯誤的資訊（例如誤記了地址），維運人員可以直接打開對應的 `.md` 檔案進行修改。修改後，AI 在下一輪對話載入時會自動採用新資訊。

### 2.3 檔案範例
```markdown
**設備型號**：Philips Alpha
**居住環境**：大樓公寓
**聯絡地址**：台北市信義區松仁路 100 號 12 樓
**聯絡電話**：0912-345-678
```

## 3. 故障排查 (Troubleshooting)

### 3.1 訊息防抖層 (Debounce)
*   **症狀**：使用者傳送訊息後，後台沒反應。
*   **檢查**：查看 `app.py` 日誌是否印出 `[收到訊息]` 但未印出 `[開始處理]`。這可能是因為 `buffer_wait` 秒數內持續有新訊息傳入導致計時器不斷重置。

### 3.2 記憶壓縮失敗
*   **症狀**：回覆速度變慢，且 Log 顯示 Token 消耗極高。
*   **檢查**：確認 `manage_memory` 節點是否有印出 `summarized` 標記。若無，請檢查 `config.toml` 的 `max_messages_threshold` 設定。

### 3.3 工具呼叫死循環
*   **症狀**：Log 重複出現 `tool_node` 呼叫同一個工具。
*   **檢查**：通常是因為工具回傳「找不到資料」，但 Agent 的 Prompt 要求「必須找到為止」。請檢查 Agent Prompt 是否包含「找不到時建議轉接真人」的邏輯。

## 4. 環境變數清單 (.env)

| 變數名 | 必要性 | 用途 |
|-------|-------|-----|
| `GEMINI_API_KEY` | 選填 | 使用 Google Gemini API 時必要 |
| `VERTEX_PROJECT_ID` | 選填 | 使用 Google Vertex AI 時必要 |
| `LINE_CHANNEL_SECRET` | 必要 | LINE Webhook 驗證 |
| `LINE_CHANNEL_ACCESS_TOKEN` | 必要 | LINE 訊息傳送 |
| `ORDER_API_URL` | 必要 | 訂單查詢 API 實體位置 |
