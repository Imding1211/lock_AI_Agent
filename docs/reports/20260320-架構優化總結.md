# 架構優化總結報告 — 2026-03-20

本報告總結近期三項重要架構升級：MMR 檢索演算法、Agentic RAG 多步重試策略、以及 Receptionist Tool-less Agent。

---

## 1. 檢索升級：MMR 演算法

### 問題

原先 pgvector 檢索使用 `similarity_search`，回傳的候選文件經常高度重複——多筆結果指向相同段落的不同切片，導致 Agent 缺乏足夠的多角度資訊來回答複雜問題。

### 解決方案

將底層檢索從 `similarity_search` 升級為 `max_marginal_relevance_search`（MMR）：

- **fetch_k = top_k × 3**：先撈出三倍候選文件
- **top_k 提升至 6**：最終回傳 6 筆兼顧相關性與多樣性的結果
- MMR 演算法在候選文件中進行二次篩選，平衡「與查詢的相關性」和「候選之間的多樣性」

### 效果

- 候選文件不再高度重複，涵蓋更多面向的資訊
- 搭配 Agentic RAG 多步重試，Agent 有更豐富的素材進行分析

## 2. Agentic RAG：Prompt 驅動自主重試

### 問題

純向量搜尋對專有名詞（品牌名、型號）與長句查詢的命中率偏低。LLM 的預設行為是一次 Tool Call 搜尋失敗就直接放棄，回覆「查無資料」。

### 解決方案

在三個專業 Agent 的 Prompt 中加入 `## 檢索策略 (Agentic Retrieval Strategy)` 區塊，教導 LLM 三項技巧：

1. **拆解與簡化關鍵字**：提煉使用者長問句中的核心搜尋詞，而非整句丟入向量搜尋
2. **強制多步重試 (Multi-step Retry)**：首次檢索失敗時，必須更換關鍵字發動第二次、甚至第三次 Tool Call
3. **同義詞盲區思考**：找「權限」找不到就改找「管理」；找「加入家人」找不到就改找「添加成員」

### 適用範圍

| Agent | 適用 | 說明 |
|-------|------|------|
| `product_expert` | 是 | 關鍵字拆解著重「產品名 + 功能」 |
| `troubleshooter` | 是 | 關鍵字拆解著重「品牌 + 型號 + 核心症狀」 |
| `youtuber` | 是 | 關鍵字拆解著重「UI 文字 / 按鈕名稱」 |
| `receptionist` | 否 | Tool-less Agent，無檢索能力 |

### 效果

- 搭配 MMR 演算法，顯著提升專有名詞與複雜問句的命中率
- Agent 不再輕易放棄，會主動嘗試多種關鍵字組合

## 3. Receptionist Agent：Tool-less 兜底分流

### 問題

使用者發送一般問候（「你好」）、感謝（「謝謝」）、或更新個資（「我的地址改了」）時，Router 因為沒有合適的意圖分類，常將這些訊息錯誤路由至訂單專員或觸發轉接真人，造成不必要的工具呼叫和糟糕的使用者體驗。

### 解決方案

新增 `general_reception` 意圖與 `receptionist` Agent：

- **意圖定義**：`general_reception` — 一般問候、感謝、個資更新等不需查閱知識庫的對話
- **Agent 設定**：`tools = []`（Tool-less），不觸發任何知識庫搜尋
- **Prompt 設計**：receptionist 以「前台接待專員」自居，負責禮貌應對、確認個資更新、引導使用者提出具體問題

### 多層漏斗防護架構

```
Guardrail 敏感詞攔截
    ↓ (通過)
Out of Domain 禮貌拒絕
    ↓ (非領域外)
Receptionist 一般接待
    ↓ (有具體問題)
專業 Agent (product_expert / troubleshooter / youtuber / order_clerk / web_researcher)
```

### 效果

- 解決「使用者更新地址卻被轉給訂單專員或真人」的錯誤路由問題
- 一般問候不再觸發不必要的知識庫搜尋，節省 API 成本與回應時間
- 形成完整的多層漏斗防護，每一層各司其職
