### 2. 轉接真人策略

Agent 遵循「自主解決優先」原則，資訊不足時的處理順序：

1. 先提供能找到的部分資訊
2. 嘗試用不同關鍵字或角度重新檢索
3. 根據專業知識提供通用建議或常見做法
4. 建議使用者提供更具體的資訊
5. 只有當使用者明確表示以上方式都無法滿足需求時，才詢問是否需要轉接真人客服

`transfer_to_human` 工具僅在以下情況使用：
- 使用者明確堅持要求轉接真人客服
- 涉及安全風險（門鎖無法上鎖、疑似被破壞）

### 2.5 Agentic RAG — 多步重試檢索策略

純向量搜尋遇到專有名詞或長句時容易「查無資料」就放棄。為解決此問題，專業 Agent 的 Prompt 中內建了 `## 檢索策略 (Agentic Retrieval Strategy)` 區塊，強制 LLM 在放棄前主動嘗試多種關鍵字組合：

1. **拆解與簡化關鍵字**：提煉使用者長問句中的核心搜尋詞，而非整句丟入向量搜尋。
2. **多步重試 (Multi-step Retry)**：首次檢索失敗時，Agent 必須更換關鍵字發動第二次、甚至第三次 Tool Call。
3. **同義詞盲區思考**：找「權限」找不到就改找「管理」；找「加入家人」找不到就改找「添加成員」。

適用 Agent：`hardware_technician`、`sales_representative`、`store_assistant`、`app_specialist`、`manual_librarian`。
不適用：`receptionist`（tool-less Agent，無檢索能力）。

此策略搭配底層 MMR 演算法（提供更多樣的候選文件），可顯著提升專有名詞與複雜問句的命中率。

> **轉接後的上下文隔離**：`merge_answers` 在 `topic_resolved` 時清除所有 agent 產出的訊息（tool_calls、ToolMessage、純文字 AI 道歉語），防止殘留的轉接脈絡影響後續 router 分類。此外，`_extract_recent_pairs` 會自動過濾尾端未配對的 HumanMessage（無對應 AI 回覆），確保 router 不會因歷史中的孤立轉接請求而誤判新意圖。
