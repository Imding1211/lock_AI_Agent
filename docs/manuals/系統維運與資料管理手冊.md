# 系統維運與資料管理手冊

本手冊旨在指導維運人員如何管理系統持久化資料（PostgreSQL / SQLite 與 User Profiles），以及在生產環境中進行故障排查。

---

## 0. 資料持久化架構總覽

系統採用「雙軌持久化」設計，三類資料各自獨立運作，具有不同的用途、備份頻率與清理策略。目前預設使用 **PostgreSQL**（可透過 `config.toml` 回退至 SQLite）：

| 資料 | 儲存位置 | 用途 | 備份建議 |
|------|---------|------|---------|
| 對話記憶（Checkpointer） | PostgreSQL `checkpoint_*` 表 | LangGraph Checkpointer 狀態（對話記憶 + 語意摘要壓縮） | 每週 `pg_dump` |
| 審計日誌（Audit Log） | PostgreSQL `audit_log` 表 | 不可變審計軌跡，記錄 `user_raw` / `user` / `ai` 原文 | 每日 `pg_dump` |
| `*.md` 輪廓檔 | `data/profiles/{user_id}.md` | 使用者軟性輪廓（偏好、過去問題、個性等） | 隨對話記憶一併備份 |
| 使用者事實 | PostgreSQL `user_facts` 表 | 硬事實（電話、地址、設備型號/品牌）SCD Type 2 版本控制 | 每週 `pg_dump` |

**設計原則**：

- 對話記憶服務於 LangGraph 的狀態管理與記憶壓縮，資料可被摘要取代後清除
- 審計日誌為不可變的審計紀錄，保留完整原文，供事後審計與分析
- `data/profiles/` 為小型 Markdown 文字檔，磁碟壓力極低
- 三者獨立運作，任一損壞不影響其他兩者

---

## 1. 資料庫管理 (PostgreSQL)

系統使用 PostgreSQL 存儲所有使用者的對話狀態與 Checkpoints（由 `langgraph-checkpoint-postgres` 管理）。

### 1.1 連線設定
透過環境變數 `POSTGRES_URI` 提供連線字串（例：`postgresql://user:pass@host:5432/dbname`），`config.toml` 的 `[memory].postgres_uri_env` 指定變數名稱。

### 1.2 資料備份與清理
*   **備份**：使用 `pg_dump` 匯出資料庫。建議每週定時備份。
    ```bash
    pg_dump -U user -h host dbname > backup/chat_history_$(date +%Y%m%d).sql
    ```
*   **清理**：
    *   **全量清理**：清空 checkpoint 相關表後重啟服務。這會導致所有使用者失憶。
    *   **特定用戶清理**：若需重置特定使用者的記憶：
        ```sql
        DELETE FROM checkpoints WHERE thread_id = 'smart_lock_{user_id}';
        DELETE FROM checkpoint_writes WHERE thread_id = 'smart_lock_{user_id}';
        ```

### 1.3 注意事項
*   確保 PostgreSQL 連線池設定適合併發量。
*   如果資料表異常增大，請檢查 `max_messages_threshold` 是否設定過大，導致壓縮頻率不足。

### 1.4 回退至 SQLite
若需回退至 SQLite，修改 `config.toml`：
```toml
[memory]
type        = "sqlite"
sqlite_path = "./data/db/chat_history.db"
```

### 1.5 使用者事實管理 (user_facts)

系統將使用者硬事實（電話、地址、設備型號/品牌）儲存於 `user_facts` 表，採用 SCD Type 2 版本控制。

**查詢當前事實**：
```sql
SELECT attr_key, attr_val FROM user_facts
WHERE user_id = '{user_id}' AND is_current = TRUE;
```

**查詢歷史事實**：
```sql
SELECT attr_key, attr_val, start_date, end_date
FROM user_facts
WHERE user_id = '{user_id}'
ORDER BY attr_key, start_date DESC;
```

**手動修正事實**：
```sql
-- 1. 過期舊值
UPDATE user_facts SET is_current = FALSE, end_date = NOW()
WHERE user_id = '{user_id}' AND attr_key = 'phone' AND is_current = TRUE;

-- 2. 插入正確值
INSERT INTO user_facts (user_id, attr_key, attr_val, is_current, start_date)
VALUES ('{user_id}', 'phone', '0912-345-678', TRUE, NOW());
```

**清理特定使用者的所有事實**：
```sql
DELETE FROM user_facts WHERE user_id = '{user_id}';
```

## 2. 審計日誌管理 (Audit Log)

系統使用 PostgreSQL 的 `audit_log` 表記錄所有使用者與 AI 的對話原文，供事後審計與分析。

### 2.1 連線設定
與對話記憶共用 `POSTGRES_URI` 環境變數，`config.toml` 的 `[storage].postgres_uri_env` 指定變數名稱。表結構會在服務啟動時自動建立（`CREATE TABLE IF NOT EXISTS`）。

### 2.2 角色欄位 (role)

| role | 說明 | 記錄時機 |
|------|------|---------|
| `user_raw` | 使用者原始碎裂訊息 | webhook 收到每條訊息時立即記錄（debounce 之前） |
| `user` | 合併後的使用者訊息 | debounce 結束後，送入 LangGraph 前記錄 |
| `ai` | AI 回覆 | LangGraph 執行完畢後記錄 |

> 連發 3 條訊息的預期紀錄：3 筆 `user_raw` + 1 筆 `user`（合併後）+ 1 筆 `ai`。

### 2.3 查看日誌

```bash
# 查看最近 30 筆紀錄（預設）
python scripts/view_logs.py

# 查看最近 50 筆紀錄
python scripts/view_logs.py 50
```

**輸出格式**：以反向時序排列（最新在最下方），每筆紀錄包含 ID、時間、角色標記、內容。

**角色標記與顏色**：

| 標記 | 對應 role | 終端機顏色 | 說明 |
|------|----------|-----------|------|
| `[RAW]` | `user_raw` | 灰色 | webhook 原始碎裂訊息（debounce 之前） |
| `[USER]` | `user` | 藍色 | 合併後送入 LangGraph 的訊息 |
| `[AI]` | `ai` | 綠色 | AI 回覆 |

也可直接使用 SQL 查詢（透過 `psql` 連入 PostgreSQL）：

```sql
-- 查看特定使用者的對話紀錄
SELECT * FROM audit_log WHERE user_id = '{user_id}' ORDER BY id DESC LIMIT 20;

-- 統計各角色的紀錄數量
SELECT role, COUNT(*) FROM audit_log GROUP BY role;
```

### 2.4 資料備份與清理
*   **備份**：使用 `pg_dump` 匯出，或單獨匯出 `audit_log` 表。
    ```bash
    pg_dump -U user -h host -t audit_log dbname > backup/audit_log_$(date +%Y%m%d).sql
    ```
*   **全量清理**：`TRUNCATE audit_log;` 後重啟服務。
*   **特定用戶清理**：
    ```sql
    DELETE FROM audit_log WHERE user_id = '{user_id}';
    ```

## 3. 使用者輪廓管理 (User Profiles)

輪廓以 Markdown 格式儲存，方便人工閱讀與微調。

### 3.1 儲存路徑
位於 `data/profiles/` 資料夾下，檔名為 `{user_id}.md`。

### 3.2 手動修正
若 AI 萃取了錯誤的資訊（例如誤記了地址），維運人員可以直接打開對應的 `.md` 檔案進行修改。修改後，AI 在下一輪對話載入時會自動採用新資訊。

### 3.3 檔案範例
```markdown
**設備型號**：Philips Alpha
**居住環境**：大樓公寓
**聯絡地址**：台北市信義區松仁路 100 號 12 樓
**聯絡電話**：0912-345-678
```

## 4. 故障排查 (Troubleshooting)

### 4.1 訊息防抖層 (Debounce)
*   **症狀**：使用者傳送訊息後，後台沒反應。
*   **檢查**：查看 `app.py` 日誌是否印出 `[收到訊息]` 但未印出 `[開始處理]`。這可能是因為 `buffer_wait` 秒數內持續有新訊息傳入導致計時器不斷重置。

### 4.2 記憶壓縮失敗
*   **症狀**：回覆速度變慢，且 Log 顯示 Token 消耗極高。
*   **檢查**：確認 `manage_memory` 節點是否有印出 `summarized` 標記。若無，請檢查 `config.toml` 的 `max_messages_threshold` 設定。

### 4.3 工具呼叫死循環
*   **症狀**：Log 重複出現 `tool_node` 呼叫同一個工具。
*   **檢查**：通常是因為工具回傳「找不到資料」，但 Agent 的 Prompt 要求「必須找到為止」。請檢查 Agent Prompt 是否包含「找不到時建議轉接真人」的邏輯。

---

### 4.4 審計日誌異常
*   **症狀**：`audit_log.db` 中缺少 `user_raw` 紀錄，只有 `user` 和 `ai`。
*   **檢查**：確認 `app.py` webhook 迴圈中的 `audit_storage` 是否為 `None`。若 startup 初始化失敗，會導致日誌靜默跳過。檢查 startup 日誌是否有 `[*] 初始化審計日誌模組` 輸出。

## 5. 環境變數清單 (.env)

| 變數名 | 必要性 | 用途 |
|-------|-------|-----|
| `LINE_CHANNEL_SECRET` | 必要 | LINE Webhook 簽章驗證 |
| `LINE_CHANNEL_ACCESS_TOKEN` | 必要 | LINE 訊息傳送（Reply / Push API） |
| `GEMINI_API_KEY` | 條件必要 | `[llm].provider = "gemini"` 時必要 |
| `VERTEX_PROJECT_ID` | 條件必要 | `[llm].provider = "vertexai"` 時必要 |
| `VERTEX_LOCATION` | 條件必要 | `[llm].provider = "vertexai"` 時必要（例: `us-central1`） |
| `OLLAMA_BASE_URL` | 選填 | Ollama 服務連線網址，預設 `http://localhost:11434` |
| `ORDER_API_URL` | 選填 | 訂單查詢 API 實體位置，優先於 `config.toml` 的 `endpoint` |
| `ORDER_API_TOKEN` | 選填 | 訂單查詢 API 的 Bearer Token |
| `POSTGRES_URI` | 條件必要 | `[memory].type` 或 `[storage].type` 為 `"postgres"` 時必要 |

## 6. 資料保留與歸檔策略

### 6.1 備份頻率建議

| 資料 | 建議頻率 | 說明 |
|------|---------|------|
| `audit_log` 表 | 每日備份 | 審計紀錄為不可變資料，遺失無法重建 |
| Checkpointer 表 | 每週備份 | 對話記憶可透過重新對話逐步恢復，但重建成本高 |
| `data/profiles/` | 隨 Checkpointer 一併備份 | 小型文字檔，磁碟壓力極低 |

使用 `pg_dump` 備份 PostgreSQL：

```bash
pg_dump -U user -h host dbname > backup/db_$(date +%Y%m%d).sql
cp -r data/profiles/ backup/profiles_$(date +%Y%m%d)/
```

### 6.2 磁碟空間監控

建議設定監控門檻（透過 PostgreSQL 查詢表大小）：

```sql
SELECT relname AS table, pg_size_pretty(pg_total_relation_size(relid)) AS size
FROM pg_catalog.pg_statio_user_tables
ORDER BY pg_total_relation_size(relid) DESC;
```

| 表 | 警告門檻 | 危險門檻 | 處理方式 |
|------|---------|---------|---------|
| `audit_log` | 500MB | 1GB | 歸檔舊紀錄 |
| Checkpointer 表 | 200MB | 500MB | 檢查 `max_messages_threshold` 是否過大 |
| pgvector collection 表 | — | — | 透過上方 SQL 查詢監控 |

### 6.3 歸檔方式

當 `audit_log` 表過大時，可匯出舊紀錄後清理：

```bash
# 1. 匯出 90 天前的紀錄至 CSV
psql -U user -h host dbname -c "\COPY (SELECT * FROM audit_log WHERE timestamp < '2026-01-01') TO 'audit_archive_20260101.csv' WITH CSV HEADER"

# 2. 刪除已歸檔的紀錄
psql -U user -h host dbname -c "DELETE FROM audit_log WHERE timestamp < '2026-01-01';"

# 3. 回收磁碟空間
psql -U user -h host dbname -c "VACUUM audit_log;"
```

Checkpointer 表的清理方式類似，但需注意：清除特定 thread 的 checkpoint 會導致該使用者的對話記憶重置。

### 6.4 Profiles 管理

`data/profiles/` 目錄下的使用者輪廓為小型 Markdown 文字檔（通常每個檔案不超過 1KB），磁碟壓力極低，一般不需要特別清理。若有不再活躍的使用者，可直接刪除對應的 `.md` 檔案。
